Based on your codebase, you already have a head start — you have RestaurantSwitcher, role-based auth, and
  per-restaurant data. Here's the roadmap:

  ---
  Multi-Tenancy (Most Critical)

  Your current model likely has restaurants scoped per user. For SaaS you need every table in Prisma scoped to an
  Organization (tenant). Two approaches:

  - Shared database, shared schema (simplest) — add organizationId to every table, enforce it in every query. You're
  probably close to this already.
  - Shared database, separate schema — one Postgres schema per tenant. Better isolation, more ops complexity.

  For a restaurant SaaS, shared schema with row-level scoping is the right call at this stage.

  Immediate Code Changes

  1. Prisma — add Organization model, add organizationId to every model that doesn't have it, add a middleware that
  auto-injects the tenant filter on every query
  2. Auth — store organizationId on the session/JWT, middleware validates tenant access on every request
  3. Stripe — Organization has a stripeCustomerId, subscriptionStatus, planTier
  4. Onboarding — a /setup flow after signup: create org → set restaurant name/timezone → invite first staff member

  ---
  The One Thing That Breaks Most SaaS Apps

  Tenant data leakage — a bug where user A sees user B's data. Guard against this by:
  - A Prisma middleware that throws if a query hits a multi-tenant table without organizationId
  - Never trusting organizationId from the client — always read it from the server session
